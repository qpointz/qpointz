## Problem Statement

When using `mill-py` or the JDBC driver, failures are often surfaced as generic
`Internal Server Error (500)` messages with insufficient context for users and operators.
This makes troubleshooting slow and hides actionable diagnostics already available (or
potentially available) in server-side exception paths.

## Goal

Establish a consistent, safe, and structured error contract so both Python and JDBC clients
surface meaningful failure details, not only generic status text.

Target diagnostic fields to preserve and expose:

- `status`
- `type` (or stable machine-readable code)
- `title`
- `detail`
- `traceId`

Scope includes parity between HTTP and gRPC client paths.

## Findings (Current State)

### HTTP server side (data service)

- Data HTTP endpoints are implemented in
  `data/services/mill-data-http-service/src/main/java/io/qpointz/mill/data/backend/access/http/controllers/AccessServiceController.java`.
- Request/response conversion is centralized in
  `data/services/mill-data-http-service/src/main/java/io/qpointz/mill/data/backend/access/http/components/MessageHelper.java`.
- No `@ControllerAdvice` / `@RestControllerAdvice` currently exists in `mill-data-http-service`,
  so exceptions can bubble to generic framework-level 500 responses.

### gRPC server side (data service)

- A gRPC advice exists:
  `data/services/mill-data-grpc-service/src/main/java/io/qpointz/mill/data/backend/MillGrpcServiceExceptionAdvice.java`.
- It mostly passes through `StatusRuntimeException` / `StatusException`, and maps `Error` to
  `Status.INTERNAL` with description.
- There is no broader structured, cross-cutting error-enrichment policy here yet.

### Python client

- Error mapping exists in `clients/mill-py/mill/exceptions.py`.
- HTTP mapping currently prefers legacy keys (`message`, `error`, `path`, `timestamp`) and can
  still collapse to generic text when payload shape differs.
- HTTP transport paths:
  - `clients/mill-py/mill/_transport/_http.py`
  - `clients/mill-py/mill/aio/_transport/_http.py`

### JDBC client

- HTTP client throws generic failure messages in
  `clients/mill-jdbc-driver/src/main/java/io/qpointz/mill/client/HttpMillClient.java`
  (e.g., status code + reason phrase), with no robust structured error-body extraction.
- Existing tests verify current generic behavior in
  `clients/mill-jdbc-driver/src/test/java/io/qpointz/mill/client/HttpMillClientTest.java`.

## In Scope

1. Define and document canonical error payload contract for HTTP responses (Problem Details-compatible).
2. Add centralized HTTP exception mapping in `mill-data-http-service`.
3. Extend Python client error parsing and propagation for Problem Details fields.
4. Extend JDBC client error parsing and exception messages to include backend details.
5. Align gRPC behavior with HTTP semantics at the client surface level.
6. Add/adjust tests for server and both clients.

## Out of Scope

- Broad redesign of domain exception taxonomy across all modules.
- Full observability platform rollout (tracing backend, dashboards, alerting).
- UI-specific error rendering redesign (covered by separate UI work items).

## Proposed Error Contract

HTTP error responses should be Problem Details-compatible (`application/problem+json`),
with at least:

- `status` (HTTP status)
- `type` (stable URI/string code)
- `title` (short summary)
- `detail` (safe actionable message)
- `traceId` (correlation token)

Compatibility note:

- Clients must continue parsing legacy envelopes (`message`, `error`, `path`, `timestamp`)
  for backward compatibility during rollout.

## Implementation Plan

1. **Server contract and mapping matrix**
   - Define mapping from common exception categories (validation, not found, parse, backend, unexpected)
     to status + `type` + `title` + `detail`.
2. **Data HTTP centralized handling**
   - Introduce `@RestControllerAdvice` in `mill-data-http-service`.
   - Ensure `MessageHelper` parse/serialization errors produce structured responses.
3. **Python client enrichment**
   - Parse Problem Details keys and preserve them in exception objects.
   - Keep raw body and request/response context for diagnostics.
   - Add gRPC error details parity where metadata/details are available.
4. **JDBC client enrichment**
   - Parse non-2xx response body JSON (with text fallback).
   - Include key fields (`status`, `type/code`, `detail`, `traceId`) in thrown exceptions.
   - Align gRPC-side detail propagation with HTTP behavior.
5. **Tests and docs**
   - Add unit/integration coverage for mapping and fallback behavior.
   - Document examples for Python/JDBC consumers.

## Acceptance Criteria

- HTTP server errors from data endpoints use structured Problem Details-compatible payloads.
- Python client exceptions show meaningful backend detail (not just `HTTP 500`) and expose structured fields when present.
- JDBC exceptions include backend-provided details/codes/traceId when available.
- HTTP and gRPC client behavior is aligned for equivalent backend failures.
- Backward compatibility retained for legacy error envelope parsing.

## Test Plan (to be executed during implementation)

### Server

- Add tests for mapped exception categories and expected status/body fields.
- Verify content type and payload shape for error responses.

### Python

- Unit tests for Problem Details parsing (`type/title/detail/status/traceId`).
- Regression tests for legacy envelope parsing and non-JSON fallback.

### JDBC

- Mock HTTP tests for 4xx/5xx with Problem Details payloads.
- Assertions on thrown exception message/cause content.

### gRPC parity

- Tests asserting that analogous backend failures produce comparable client-level error context.

## Risks and Mitigations

- **Risk:** leaking sensitive internal details in `detail`.  
  **Mitigation:** define safe message policy and keep deep stack traces server-side logs only.

- **Risk:** contract drift between services/modules.  
  **Mitigation:** centralize mapping policy in data service and document shared semantics.

- **Risk:** client breakage due to strict parsing assumptions.  
  **Mitigation:** parser supports both Problem Details and legacy envelopes with text fallback.
