# WI-394 - Analysis SQL apply and editor history

## Goal

Implement Analysis copilot handling for SQL proposals from inline chat.

## Requirements

- Extend `hostIntegrations.ts` into the typed inline host action registry defined by WI-389:
  - register one Analysis handler for `inline-analysis`
  - dispatch `sql.apply`, `sql.applyAndRun`, and `sql.copy` actions from artifact strips
  - keep all Analysis mutations and async execution work inside the Analysis host handler
  - return handled/unhandled status for UI fallback where useful
- Replace Analysis SQL application paths that bypass artifact strip actions:
  - remove or disable unconditional SQL auto-apply effects for Analysis artifacts
  - remove or supersede markdown SQL extraction as an editor mutation path
  - keep all Analysis SQL mutations behind artifact strip events and settings interpretation
- Add Analysis copilot setting controls that appear only when an inline chat session exists for
  the current Analysis context:
  - Auto-apply SQL proposals (`autoapply.enabled`)
  - Run after applying (`autorun.enabled`)
- Analysis handles SQL artifact strip actions:
  - `Apply` updates the SQL editor.
  - `Apply & Run` updates the SQL editor and executes the resulting query.
- Apply & Run execution policy:
  - close any active execution session before starting the new run
  - reset pagination to page 1/default page size
  - run the full applied SQL document, not a selected fragment
  - mark the editor dirty when applied SQL differs from the saved snapshot
  - disable Apply & Run while `isExecuting`
  - keep the applied SQL in the editor if execution fails
  - show failures with the same error pattern used by SQL artifacts / query execution
- Preserve the expectation that chat-generated SQL is validated by the backend capability where
  validation is available.
- Analysis interprets settings:
  - when `autoapply.enabled` is on, the first SQL proposal in a turn is applied to the editor
    automatically
  - when `autorun.enabled` is on and the first SQL proposal in a turn is applied, the query
    executes automatically
  - later SQL proposals in the same turn are never auto-applied or auto-run; users may still apply
    them manually from their artifact strips
  - `autorun.enabled` never runs a proposal that has not been applied
- Implement the Analysis behavior matrix:
  - auto-apply off, auto-run off: show proposal; user chooses Apply or Apply & Run
  - auto-apply off, auto-run on: show proposal; user must apply; Apply also runs
  - auto-apply on, auto-run off: apply proposal automatically; user runs manually
  - auto-apply on, auto-run on: apply proposal automatically, then run
- Remove or replace any existing automatic SQL artifact apply behavior for `inline-analysis`.
- Existing prose-only chat responses must not mutate the SQL editor through markdown code-fence
  extraction.
- Add editor-level undo/redo support:
  - history is UI/session-level only and is not persisted.
  - history resets after Save.
  - chat-applied SQL participates in the same in-session history model as normal editor edits.
  - toolbar controls expose Undo and Redo when available.
  - keyboard/editor behavior remains compatible with current SQL editing.
- Preserve existing dirty-state behavior and unsaved-change prompts.
- Preserve existing saved query load, save, delete, and chat-to-analysis handoff behavior.

## Acceptance Criteria

- SQL generated by the Analysis copilot does not replace the editor until the user applies it.
- Prose-only responses and fenced SQL in markdown do not mutate the editor unless represented as an
  actionable SQL artifact strip.
- By default, auto-apply and auto-run are off.
- Analysis setting controls are hidden until an inline chat exists for the active Analysis context.
- The settings are labeled "Auto-apply SQL proposals" and "Run after applying".
- Turning on auto-apply causes the next SQL proposal to update the editor automatically.
- When multiple SQL proposals arrive in one turn, auto-apply targets only the first proposal.
- When multiple SQL proposals arrive in one turn, auto-run targets only the first proposal after it
  has been applied.
- Later SQL proposals in the same turn remain manually actionable.
- Turning on auto-run executes the query only after the proposal has been applied.
- Apply updates SQL and marks the editor dirty when it differs from the saved snapshot.
- Apply & Run updates SQL and starts query execution using the applied SQL.
- Apply & Run closes the previous execution session, resets pagination, and runs the full applied
  SQL document.
- Apply & Run is disabled while `isExecuting`.
- If Apply & Run fails, the applied SQL remains in the editor and the failure is shown in the
  existing Analysis/query error pattern.
- Undo returns to the previous SQL version after a chat apply.
- Redo reapplies the chat-generated SQL after undo.
- Undo/redo history is in-memory UI/session state only and resets after Save.
- Existing Analysis query execution and pagination behavior are unchanged.
